# Release v1.1.6

**Release Date:** 2026-05-11

## Bug Fix: Date Range Filtering Now Works Correctly

### Problem

When using `startdate` and `enddate` with a `YYYY-MM-DD` string (e.g. `"2026-05-11"`), both parameters were resolved to **midnight (00:00:00)** of that day. This created a zero-length time window, causing the Withings API to return empty results — even when measurements existed for that date.

Affected tools: `get_measurements`, `get_sleep_details`, `get_heart_rate`

### Fix

`enddate` strings in `YYYY-MM-DD` format are now resolved to **23:59:59** of that day instead of 00:00:00, so the full day is included in the query window.

Unix timestamps passed as `enddate` are not affected — they are forwarded as-is.

### Changes

Modified `src/withings_mcp_server/server.py`:

- Added `end_of_day: bool = False` parameter to `_parse_date()`
- `_get_measurements`, `_get_sleep_details`, `_get_heart_rate` now pass `end_of_day=True` for `enddate`

### Before / After

```
startdate: "2026-05-11"  →  2026-05-11 00:00:00 local  ✅ unchanged
enddate:   "2026-05-11"  →  2026-05-11 00:00:00 local  ❌ before (zero window)
enddate:   "2026-05-11"  →  2026-05-11 23:59:59 local  ✅ after  (full day)
```
